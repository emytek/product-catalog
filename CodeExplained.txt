================================================================================
  1. server.js — Entry Point, Clustering, Graceful Shutdown
================================================================================

LINE:  const NUM_WORKERS = config.app.isProduction ? os.cpus().length : 1;

  This decides how many Node.js processes to run.
  - In PRODUCTION: one process per CPU core (e.g. 4 cores = 4 workers).
    Each worker is a full copy of the app running independently.
    If one core is busy, the OS sends the next request to a free worker.
    This is called "horizontal scaling" — you multiply capacity without
    changing any code.
  - In DEVELOPMENT: only one worker, so logs are easier to read.


LINE:  if (cluster.isMaster && NUM_WORKERS > 1) { ... cluster.fork() ... }

  Node.js has a built-in "cluster" module.  One process is the "Master" —
  it never handles HTTP requests itself.  It only:
    1. Spawns (forks) the worker processes.
    2. Watches them for crashes and restarts them automatically.

  The fork() call is like making a copy of the entire running program.
  Each copy shares the same port (3000) — the OS distributes incoming
  connections across all copies.


LINE:  cluster.on("exit", (worker) => { cluster.fork(); });

  If any worker crashes (unhandled error, out of memory, etc.), this line
  immediately spawns a replacement.  The app stays alive even when one
  process dies — this is called "self-healing".


LINE:  await getPool();   // Pre-warms the connection pool

  Before the HTTP server opens, we connect to the database.
  "Pre-warming" means the first real user request does not have to wait for
  the TCP connection + SQL Server login to complete.  It is already done.


LINE:  server.close(async () => { await closeQueues(); await closePool(); process.exit(0); });

  "Graceful shutdown" — when Cloud Foundry (BTP) wants to stop the app
  (e.g. scaling down, deploying a new version), it sends the SIGTERM signal.
  Instead of cutting off instantly, we:
    Step 1: Stop accepting new HTTP connections (server.close).
    Step 2: Wait for requests already in-flight to finish.
    Step 3: Drain the background job queue (closeQueues).
    Step 4: Close all database connections cleanly (closePool).
    Step 5: Exit with code 0 (success).
  This prevents users mid-request from seeing an error.


LINE:  setTimeout(() => { process.exit(1); }, 10000).unref();

  A safety net: if graceful shutdown hangs (e.g. a database query that never
  finishes), this forces the process to exit after 10 seconds.
  ".unref()" tells Node not to keep the process alive just for this timer —
  it only fires if shutdown is still running when time runs out.


LINE:  process.on("unhandledRejection", (reason) => { logger.error(...) });

  Every Promise in Node.js can reject (fail).  If a rejection is never
  caught with .catch() or try/catch, it becomes "unhandled".
  Without this line, Node would silently ignore those errors (or crash in
  newer versions).  This line logs every unhandled rejection so you can debug it.


================================================================================
  2. src/app.js — Express Setup and Middleware Stack
================================================================================

WHAT IS MIDDLEWARE?
  Middleware is code that runs on every HTTP request before (and after) your
  route handlers.  Think of it as a pipeline — each piece of middleware does
  one job, then passes the request to the next piece.
  Order matters: the first app.use() call runs first.


LINE:  require("express-async-errors");

  Express was originally written before JavaScript had async/await.
  If an async route handler throws an error, Express's default behaviour does
  NOT catch it — the error disappears and the request hangs forever.
  This one-line import patches Express so async errors are automatically
  forwarded to the error middleware.  Without it, errors in async functions
  would be silent and requests would time out.


LINE:  app.use(helmet({ crossOriginResourcePolicy: { policy: "cross-origin" } }));

  Helmet adds approximately 15 security-related HTTP headers automatically:
    - X-Frame-Options: DENY        — prevents your page being loaded in an iframe
                                     (clickjacking protection)
    - X-XSS-Protection             — tells browsers to block XSS attacks
    - Strict-Transport-Security    — forces HTTPS in production
    - Content-Security-Policy      — controls what resources the browser loads
  The "cross-origin" option is required because SAPUI5 loads its library from
  ui5.sap.com (a different domain).  Without it, Helmet would block those
  resources and the entire frontend would fail to load.


LINE:  app.use(cors({ origin: (origin, callback) => { ... } }));

  CORS = Cross-Origin Resource Sharing.
  Browsers block JavaScript from calling APIs on a different domain or port
  by default.  When the frontend (localhost:8080) calls the backend (localhost:3000)
  these are different "origins" (different ports).
  This middleware tells the browser: "yes, localhost:8080 is allowed to call this API".
  In production on BTP, the approuter serves both frontend and API from the same
  origin, so CORS is effectively a no-op there.


LINE:  app.use(compression({ level: 6, threshold: 512 }));

  Compresses (gzip) all HTTP responses before sending them over the network.
  - level: 6   — compression strength (1=fast/weak, 9=slow/strong, 6=balanced)
  - threshold: 512 — only compress responses larger than 512 bytes.
    No point compressing tiny responses — the overhead outweighs the gain.
  For the /products/catalog response (25 products, ~15KB JSON), gzip typically
  reduces size by 70%.  This matters on BTP where bandwidth costs money
  and mobile/slow connections are common.


LINE:  app.use(morgan(morganFormat, { stream: { write: (msg) => logger.http(msg) } }));

  Morgan is an HTTP request logger.  It writes one line per request:
    GET /api/v1/products/catalog 200 45ms
  The "stream" option routes Morgan's output through our Winston logger
  so all log output has the same format and goes to the same log file or service.
  The "skip" option prevents /health from being logged — health checks happen
  every few seconds and would flood the logs with noise.


LINE:  app.use(express.json({ limit: "1mb" }));

  Parses the request body as JSON and puts it on req.body.
  Without this, req.body would be undefined for POST and PUT requests.
  The 1mb limit prevents someone sending a 100MB JSON payload to crash the server
  (a basic protection against denial-of-service attacks).


LINE:  if (config.app.isProduction) { app.set("trust proxy", 1); }

  In production on BTP, requests pass through a load balancer before reaching
  the app.  The load balancer sets the X-Forwarded-For header with the real
  client IP address.
  "trust proxy: 1" tells Express to read that header.
  Without it, the rate limiter would see the load balancer's IP for every request
  and think all traffic comes from one source — rate limiting would be useless.


LINE:  app.use(`/api/${config.app.apiVersion}`, v1Router);

  Mounts all API routes under /api/v1.
  Using a version number in the URL (/v1) is standard practice — if you ever
  need to make breaking changes to the API, you create /v2 and clients using
  /v1 continue to work without any changes on their side.


LINE:  app.use(errorMiddleware);   // MUST be last

  The global error handler must be registered AFTER all routes.
  Express identifies error-handling middleware by the fact that it has exactly
  4 parameters (err, req, res, next).  If it were registered before the routes,
  it would never receive errors because no routes have run yet.

  3. src/config/environment.js — Environment Variable Management
================================================================================

WHAT ARE ENVIRONMENT VARIABLES?
  Configuration values (passwords, host names, ports) that change between
  environments (your laptop in development vs. production on BTP cloud).
  They live outside the code so you never hardcode secrets in source files.
  Stored in the .env file locally; BTP injects them automatically in the cloud.


LINE:  if (process.env.NODE_ENV !== "production") { require("dotenv").config(...); }

  The dotenv package reads the .env file and loads its key=value pairs into
  process.env (Node's global environment object).
  In production (BTP), we do NOT load .env — BTP injects the real values
  directly into the process.  Reading .env in production would be a mistake
  because the file likely doesn't exist there, or contains dev credentials.


LINE:  function required(key) { ... throw new Error(...) }

  "Fail fast" principle: if a required setting is missing, crash at startup
  with a clear message rather than crashing later deep inside a database query.
  Without this, a typo in the .env file would result in a confusing database
  connection failure on the first real request — hard to diagnose.


LINE:  function parseCFServices() { const vcap = process.env.VCAP_SERVICES; ... }

  On SAP BTP Cloud Foundry, when you bind a user-provided service (like your
  MSSQL database), BTP injects all connection details into a special environment
  variable called VCAP_SERVICES as a JSON string.
  This function reads that JSON, finds the "product-catalog-mssql" entry, and
  extracts host/port/database/user/password from it.
  Result: the exact same code works locally (reads .env) and on BTP (reads
  VCAP_SERVICES) with zero code changes between environments.

  4. src/config/database.js — Connection Pooling
================================================================================

WHY CONNECTION POOLING?
  Opening a database connection involves:
    - A TCP three-way handshake (network round trip)
    - SQL Server authentication (login, permission check)
    - Memory allocation on the server
  This takes 50-200ms.  If every HTTP request opened a fresh connection, a page
  with 5 API calls would spend 250-1000ms just on connection overhead.

  A "pool" pre-opens several connections at startup and keeps them alive.
  When a request needs the database, it "borrows" a ready connection from the pool,
  uses it, and returns it.  No TCP handshake needed — nearly instant.


LINE:  let _pool = null;

  A module-level variable that holds the single shared pool.
  This is the "Singleton" pattern — no matter how many files call getPool(),
  they all get the same pool object, not a new one each time.


LINE:  if (_pool && _pool.connected) { return _pool; }

  Before creating a new pool, check if one already exists and is still connected.
  This is what makes getPool() safe to call from any repository at any time —
  only the first call creates the pool; every subsequent call reuses it.


LINE:  _pool.on("error", (err) => { logger.error("[DB] Pool error:", err.message); });

  The pool runs in the background and can encounter errors at any time
  (SQL Server restarts, network blips).  Without this listener, those errors
  would become "unhandled events" that crash the Node process silently.
  By listening and logging them, the app stays alive and the issue is visible.


LINE:  pool.max: 10, pool.min: 2

  max: 10 — never hold more than 10 DB connections simultaneously.
    SQL Server has a connection limit; cloud databases often charge per connection.
    10 is safe for a 256MB BTP instance.
  min: 2 — always keep at least 2 connections pre-warmed.
    The first requests after startup don't have to wait for connections to open.


LINE:  await request.query("SELECT 1 AS alive");   // in isHealthy()

  The /health endpoint calls this to check if the database is reachable.
  "SELECT 1" is the simplest possible query — it costs almost nothing and returns
  immediately if SQL Server is responsive.
  If it throws, isHealthy() returns false, /health returns HTTP 503
  (Service Unavailable), and BTP's load balancer stops sending traffic to this
  instance until it recovers.

  5. src/config/cache.js — Two-Level Cache
================================================================================

WHY CACHING?
  The /products/catalog response involves:
    - A SQL query joining 4 tables (Products, Categories, Suppliers, Tags)
    - STRING_AGG aggregation for comma-separated tags
    - Sending results over a network connection
  This takes approximately 10-50ms.  If 100 users load the catalog every second,
  that is 1,000 to 5,000ms of database time per second — all for identical data.

  A cache stores the result in memory after the first request.  Every subsequent
  request gets the cached copy in ~0.1ms, until the cache expires or a product
  is created/updated/deleted (which invalidates the cache).


LINE:  const l1 = new NodeCache({ stdTTL: config.cache.ttlSeconds, useClones: false });

  L1 = Level 1 cache.  NodeCache stores data in the Node.js process memory.
  - stdTTL: default time-to-live (300 seconds = 5 minutes).
    After 5 minutes, the cached value is deleted and the next request re-fetches
    from the database, picking up any changes.
  - useClones: false — stores a reference to the object, not a deep copy.
    Faster, but means callers must not modify cached objects in-place.


LINE:  let redis = null;

  L2 = Level 2 cache (Redis).  Redis is a separate in-memory database.
  It starts as null — Redis is entirely optional.  If REDIS_HOST is not set in
  .env, the app uses L1 only (perfectly fine for development and small deployments).
  When Redis IS available (as a BTP service), it is shared across all CF
  instances so they all read from the same cache.


LINE:  retryStrategy: (times) => (times > 5 ? null : Math.min(times * 100, 2000))

  If Redis disconnects, ioredis automatically tries to reconnect.
  This function controls the delay before each retry:
    1st retry: 100ms, 2nd: 200ms, 3rd: 300ms, ... up to 2000ms maximum.
    After 5 failed attempts, return null = give up retrying.
  This "exponential backoff" prevents hammering an unavailable Redis server
  with hundreds of connection attempts per second.


LINE:  const l1val = l1.get(key);
       if (l1val !== undefined) return JSON.parse(l1val);

  L1 is always checked first because it is in the same process — zero network cost.
  If the value is there, we parse the stored JSON string back to a JavaScript
  object and return it immediately, without touching Redis or the database at all.


LINE:  l1.set(key, l2val);   // Promote to L1

  "Cache promotion": if L1 missed but L2 (Redis) had the data, we copy it
  into L1.  The next request for this key will hit L1 and skip Redis.
  This matters when running multiple CF instances: Instance A may have the
  catalog in its L1 but Instance B does not.  Instance B gets it from Redis
  once, promotes it to its own L1, and then serves future requests locally.


LINE:  do { const [nextCursor, found] = await redis.scan(...) } while (cursor !== "0");

  Redis SCAN iterates through all keys in small batches without blocking.
  Used in delByPrefix() to find all cache keys starting with "products:list:"
  (so all paginated list caches are cleared after a write operation).
  Why not the KEYS command?  KEYS blocks Redis (it is single-threaded) until
  it has scanned every key in the database — dangerous on a large dataset.
  SCAN is the safe, non-blocking alternative.

  6. src/config/queue.js — Async Job Queue
================================================================================

WHY QUEUES?
  When a product is created, we want to:
    - Refresh the stats cache
    - Write an audit log entry
  These do NOT need to happen before we respond to the user.
  If done synchronously, the POST /products response would be 50-100ms slower.
  By pushing them onto a queue, we respond instantly and the tasks run in
  the background.


LINE:  class InProcessQueue extends EventEmitter { ... setImmediate(async () => { ... }) }

  When Redis is not available, this fallback queue is used.
  setImmediate() schedules the job to run "on the next iteration of the event loop"
  — meaning after the current HTTP response has already been sent to the browser.
  This keeps the API fast while still executing the background tasks.
  EventEmitter is Node's built-in publish/subscribe system.


LINE:  _productQueue = new Bull("product-operations", { redis: redisOpts });

  Bull is a Redis-backed job queue library.  When Redis IS available, Bull:
    - Persists jobs to Redis (survives process restarts — no jobs lost)
    - Supports automatic retries with backoff if a job fails
    - Allows multiple worker processes to consume from the same queue
    - Provides a monitoring dashboard (Bull Board) for job status
  The InProcessQueue fallback has none of these features — it exists only
  for local development convenience when Redis is not running.


LINE:  _productQueue.on("failed", (job, err) => { logger.error(...) });

  If a background job throws an error, Bull emits this event.
  We log it so you can investigate.  The HTTP request that triggered the job
  is already complete and the user is unaffected — the job simply failed
  silently from the user's perspective, but is visible in the logs.

  7. src/repositories/BaseRepository.js — Base Class for All DB Access
================================================================================

WHAT IS A REPOSITORY?
  The Repository Pattern means: all database access code lives in one place
  called a repository.  Nothing else writes SQL.
  Services call methods like findById(), insert(), delete().
  Only the repository knows SQL.  If you change from SQL Server to PostgreSQL,
  you only rewrite the repositories — services and controllers are untouched.


LINE:  class BaseRepository { constructor(tableName) { this.tableName = tableName; } }

  All repositories (ProductRepository, CategoryRepository, SupplierRepository)
  extend this class and call super("Products") in their constructors.
  The tableName is used in log and error messages so you know which table caused
  a problem without reading the full stack trace.


LINE:  async getRequest() { const pool = await getPool(); return pool.request(); }

  A "request" is mssql's object for executing one SQL statement.
  You add parameters to it (.input()), then execute it (.query() or .execute()).
  Each request automatically borrows a connection from the pool and returns it
  when done — the repository does not manage connections manually.


LINE:  async withTransaction(callback) {
         await transaction.begin();
         const result = await callback(transaction);
         await transaction.commit();
       } catch (err) {
         await transaction.rollback();
       }

  A transaction groups multiple SQL statements so that either ALL succeed
  or NONE do.  Creating a product involves three steps:
    1. Generate a new ProductID (sp_GetNextProductID)
    2. Insert the product row
    3. Insert rows into ProductTags

  If step 3 fails, the ROLLBACK undoes steps 1 and 2.
  The database is never left in a half-finished state.

  The callback pattern lets service code look like this:
    await productRepo.withTransaction(async (tx) => {
        const id = await productRepo.generateNextId(tx);
        await productRepo.insert(record, tx);
        await productRepo.syncTags(id, tags, tx);
    });
  Clean and readable — the caller never manually writes BEGIN/COMMIT/ROLLBACK.


LINE:  if (err.number) { throw new DatabaseError(`...${err.message}`); }

  mssql errors have a "number" property (SQL Server's internal error code).
  By re-wrapping them as a DatabaseError (our custom class), the service layer
  only sees clean domain-level errors, not raw mssql library objects.
  This is the "abstraction" part of the Repository Pattern.


================================================================================
  8. src/services/product.service.js — Business Logic Layer
================================================================================

WHAT IS A SERVICE?
  The Service Layer sits between controllers (HTTP) and repositories (SQL).
  A controller receives an HTTP request and calls the service.
  The service contains all business logic — validation, orchestration,
  cache management, queue dispatch.
  The repository just executes SQL.


LINE:  const cacheKey = `products:list:${JSON.stringify({ ...query, page, limit })}`;

  Each unique combination of search/filter/sort/page parameters gets its own
  cache entry.  The key is a JSON-stringified representation of all query params.
  Example:
    products:list:{"search":"laptop","category":"Electronics","page":1,"limit":20}
  Two requests with the same filters get the same cached result.
  Two requests with different filters each get their own independent cache entry.


LINE:  const category = await categoryRepo.findByName(dto.Category);
       if (!category) throw new NotFoundError(`Category "${dto.Category}" does not exist.`);

  Business rule validation: before inserting a product, verify the category name
  typed by the user actually exists in the database.
  We throw a NotFoundError (our custom 404 error class) which the error middleware
  catches and converts into a proper JSON error response.
  Without this check, the SQL INSERT would fail with a cryptic foreign-key
  constraint error from SQL Server — confusing to the user.


LINE:  const supplierID = await supplierRepo.findOrCreate(dto.Supplier);

  "Find or create" pattern: if the supplier name already exists, return its ID.
  If it does not exist yet, insert a new row and return the new ID.
  This lets users type any supplier name in the form without needing a
  separate "create supplier" step.  The database stays normalised (no duplicate
  supplier rows) while the user experience stays simple.


LINE:  const created = await productRepo.withTransaction(async (tx) => {
           const productId = await productRepo.generateNextId(tx);
           await productRepo.insert(record, tx);
           await productRepo.syncTags(productId, dto.Tags || [], tx);
           return productId;
       });

  Everything in a transaction:
  - generateNextId calls the sp_GetNextProductID stored procedure.
    UPDLOCK and SERIALIZABLE isolation ensure two simultaneous requests cannot
    generate the same ID (no two products can both become P026 at the same time).
  - insert writes the product row.
  - syncTags deletes all existing tag links for this product, then re-inserts
    the new set.  This is simpler and safer than trying to diff old vs new tags.
  If any step throws, the transaction rolls back and nothing is saved.


LINE:  await this._invalidateWriteCaches(created);

  After any write (create/update/delete), cached data is now stale.
  We delete these cache entries:
    - "catalog"          — the full product list used by the frontend
    - "stats"            — KPI counts (TotalProducts etc. have changed)
    - "product:P026"     — this specific product's individual cache entry
    - "products:list:*"  — all paginated list caches (any page may be affected)
  The next request for any of these will re-query the database and get fresh data.


LINE:  this._enqueuePostCreateJobs(created);   // fire-and-forget

  We do NOT await this call.  The method starts the background jobs but we
  do not wait for them to finish.  The HTTP response is sent immediately after
  the line above.  The background jobs run on the next event loop iteration.
  Even if a background job fails, the user's create operation already succeeded —
  they will never see that failure.


LINE:  module.exports = new ProductService();

  We export an INSTANCE (new ProductService()), not the class itself.
  Every file that does require("./product.service") gets the exact same object.
  This is the Singleton pattern — cache key constants and any internal state
  are shared across all callers without being re-created each time.

  9. src/middleware/error.middleware.js — Global Error Handler
================================================================================

WHAT IS AN ERROR HANDLER?
  Any unhandled error in any route or middleware eventually reaches this function.
  express-async-errors ensures async errors arrive here too.
  Without a global error handler, Express would send a raw HTML error page
  or crash.  With it, every error returns a consistent JSON response.


LINE:  function errorMiddleware(err, req, res, next) { ... }   // 4 parameters

  Express identifies error-handling middleware SPECIFICALLY by having exactly
  4 parameters, with the first being the error object.
  If you accidentally write only 3 parameters, Express treats it as normal
  middleware and errors bypass it entirely.
  The 4th parameter "next" must be declared even if never called — this is an
  Express quirk with no workaround.


LINE:  if (err.isOperational) {
           logger.warn(...)    // expected error
       } else {
           logger.error(...)   // unexpected — log full stack trace
       }

  "Operational" errors are expected and normal:
    product not found (404), validation failed (422), duplicate ID (409).
  These are business events — log at WARN level (no alarm needed).

  "Programmer" errors are unexpected:
    null reference, missing argument, logic bug.
  Log at ERROR level with the full stack trace so you can find and fix the bug.


LINE:  const SQL_UNIQUE_VIOLATION = 2627;
       const SQL_FK_VIOLATION     = 547;
       const SQL_NULL_VIOLATION   = 515;

  SQL Server communicates errors via numeric codes stored in err.number.
  2627 — duplicate primary/unique key    → respond with 409 Conflict
  547  — foreign key violation           → respond with 400 Bad Request
  515  — NULL inserted into NOT NULL column → respond with 400 Bad Request
  Without this mapping, raw SQL Server error messages would reach the client,
  leaking internal schema information that attackers could exploit.


LINE:  const message = process.env.NODE_ENV === "production"
           ? "An unexpected error occurred. Please try again later."
           : err.message;

  In production, unexpected errors return a safe generic message.
  You never want stack traces or internal error details visible to end users —
  they reveal implementation details that attackers can use to craft exploits.
  In development, you see the real error message to help you debug faster.


================================================================================
  10. ARCHITECTURE SUMMARY — How All the Layers Fit Together
================================================================================

A request for PUT /api/v1/products/P022 (update a product) flows like this:

  BROWSER (SAPUI5)
    |
    | HTTP PUT http://localhost:3000/api/v1/products/P022
    | Body: { ProductName, Category, Price, Tags, ... }
    v
  app.js — MIDDLEWARE PIPELINE
    |  1. Helmet adds security headers to the response
    |  2. CORS checks the Origin header is on the allowed list
    |  3. Compression will gzip the response before it leaves
    |  4. Morgan logs the incoming request line
    |  5. express.json() parses the body JSON into req.body
    |  6. generalLimiter checks the rate limit (fails with 429 if exceeded)
    v
  products.routes.js — ROUTING
    |  Matches the PUT /:id route
    |  1. writeLimiter: stricter rate limit for write operations
    |  2. validateParams: "P022" must match /^P\d+$/ pattern
    |  3. validateBody(productUpdateSchema): Joi validates every body field
    |     (ProductName required, Price must be positive number, etc.)
    v
  product.controller.js — CONTROLLER (thin HTTP layer)
    |  Extracts req.params.id = "P022" and req.body
    |  Calls productService.updateProduct("P022", req.body)
    |  Does NOT know anything about SQL, cache, or queues
    v
  product.service.js — SERVICE (business logic)
    |  1. productRepo.findById("P022")       — throws 404 if not found
    |  2. categoryRepo.findByName("Furniture") — throws 404 if category invalid
    |  3. supplierRepo.findOrCreate("IKEA")  — inserts supplier if new
    |  4. Opens a database transaction:
    |       - productRepo.update("P022", record, tx)
    |       - productRepo.syncTags("P022", ["sale","new"], tx)
    |     Commits on success; rolls back everything on failure
    |  5. _invalidateWriteCaches("P022")  — clears stale catalog, stats, list caches
    |  6. _enqueuePostUpdateJobs("P022")  — audit-log job queued, no wait
    |  7. getProductById("P022")          — fetches fresh product for the response
    v
  product.repository.js — REPOSITORY (SQL only)
    |  Executes: UPDATE [Products] SET ProductName=@n, Price=@p ... WHERE ProductID='P022'
    |  Maps the SQL result row to a flat JavaScript DTO object
    |  Only knows about SQL Server — nothing about HTTP, caching, or business rules
    v
  database.js — CONNECTION POOL
    |  Borrows a ready connection from the pool (no TCP handshake needed)
    |  Sends the SQL statement to SQL Server
    |  Returns the connection to the pool for the next request to use
    v
  SQL SERVER

KEY DESIGN PRINCIPLES APPLIED
───────────────────────────────

  SOLID — Single Responsibility:
    Controller only handles HTTP (extract params, call service, send response).
    Service only handles business rules (validate, orchestrate, cache, queue).
    Repository only handles SQL (build query, execute, map result).
    Each file has exactly one reason to change.

  SOLID — Open/Closed:
    BaseRepository can be extended (ProductRepository, CategoryRepository)
    without modifying BaseRepository itself.  Adding a new entity means
    adding a new file, not editing existing ones.

  SOLID — Dependency Inversion:
    The service depends on the repository interface (findById, insert, etc.),
    not on the mssql library directly.  To switch from SQL Server to PostgreSQL
    you would rewrite the repositories only — the service is untouched.

  DRY — Don't Repeat Yourself:
    BaseRepository holds pool access and transaction logic once, shared by all repos.
    ApiResponse holds all JSON response formatting in one place.
    environment.js is the single source of truth for all config values.
    No file ever calls process.env directly except environment.js.

  FAIL FAST:
    required() in environment.js crashes startup immediately if a config value
    is missing — better than crashing mid-request with a confusing error.

  DEFENCE IN DEPTH:
    Helmet (security headers) + CORS + rate limiting + Joi validation +
    parameterised SQL queries (no injection possible) + error message hiding
    in production.  Multiple independent layers of protection mean one missed
    check does not compromise the entire application.

  GRACEFUL DEGRADATION:
    Redis is optional — app uses in-process cache if Redis is absent.
    Bull queue falls back to in-process EventEmitter if Redis is absent.
    The frontend falls back to local JSON if the backend API is unreachable.
    The app degrades gracefully rather than failing completely.
